Option 1 — Power BI with Azure DevOps Analytics (OData) (fastest to value) 

What you get 

Project‑ and pipeline‑level runs with duration, queue time, success/failure, and trends. 

A single semantic endpoint you can refresh on a schedule from Power BI Service. 

How to wire it up 

Enable Analytics (it’s on by default for ADO Services orgs; required permission: Analytics View or Project Collection read). 

In Power BI Desktop, Get Data → OData feed, and connect to: 

Useful tables for pipelines: 

PipelineRuns — pipeline/run facts (columns include PipelineName, Duration, QueueDuration, Result) 

BuildRuns / ReleaseDeployments (if you still have classic) 

Projects, Repositories for lookups / joins 

A colleague shared this exact approach (OData to Power BI + key tables like PipelineRuns with Duration and “YAML vs Classic” classification) as the quickest route to org‑wide stats and visuals. 2 

Tips 

For YAML vs. Classic, join PipelineRuns to pipeline definitions (or use definition.type where available in your model). 3 

Create measures for P50/P90 duration, failure rate, MTTR. 

Publish to Power BI Service; set refresh with an AAD app or service principal. 

 

Option 2 — REST API inventory + runs (full control; works for governance catalogs) 

Why this route 

You need a complete inventory of pipelines across all projects (including IDs, path, last run, auth settings, agent pool, etc.). 

You want to join in security/governance metadata not exposed in Analytics. 

API pattern 

List all projects in the org: 
GET https://dev.azure.com/{org}/_apis/projects?api-version=7.1-preview.4 

For each project, list pipelines: 
GET https://dev.azure.com/{org}/{project}/_apis/pipelines?api-version=7.1-preview.1 

For each pipeline, get recent runs (for health/status): 
GET https://dev.azure.com/{org}/{project}/_apis/pipelines/{pipelineId}/runs?api-version=7.1-preview.1 

(Classic builds) If you still have classic build/release, also use: 

Builds: GET .../{project}/_apis/build/builds?api-version=7.1 

Releases: GET .../{project}/_apis/release/releases?api-version=7.1 

These REST endpoints were called out in the same internal guidance as the way to pull detailed stats and build your own inventories. 4 

Sample script outline (PowerShell) 

1     $pat      = "<ADO_PAT>"            # scope: Read (Project & Build); consider using AAD App + OAuth where possible 

2     $org      = "contoso" 

3     $auth     = :ToBase64String([Text.Encoding]::ASCII.GetBytes(":$pat")) 

4     $headers  = @{ Authorization = "Basic $auth" } 

5      

6     $projects = (Invoke-RestMethod -Headers $headers -Uri "https://dev.azure.com/$org/_apis/projects?api-version=7.1").value 

7     $all = @() 

8      

9     foreach ($p in $projects) { 

10       $pipelines = (Invoke-RestMethod -Headers $headers -Uri "https://dev.azure.com/$org/$($p.name)/_apis/pipelines?api-version=7.1-preview.1").value 

11       foreach ($pl in $pipelines) { 

12         $runs = (Invoke-RestMethod -Headers $headers -Uri "https://dev.azure.com/$org/$($p.name)/_apis/pipelines/$($pl.id)/runs?api-version=7.1-preview.1&$top=5").value 

13         $latest = $runs | Sort-Object createdDate -Descending | Select-Object -First 1 

14         $all += [pscustomobject]@{ 

15           Project       = $p.name 

16           PipelineId    = $pl.id 

17           PipelineName  = $pl.name 

18           LatestRunId   = $latest.id 

19           LatestStatus  = $latest.state 

20           LatestResult  = $latest.result 

21           LatestQueued  = $latest.createdDate 

22           LatestDuration= (New-TimeSpan -Start $latest.createdDate -End $latest.finishedDate).TotalMinutes 

23         } 

24       } 

25     } 

26      

27     $all | Export-Csv ".\ado-pipelines-inventory.csv" -NoTypeInformation 

You can expand this to fetch checks, service connections, agent pool usage, and variable group references for deeper governance auditing.

Design your “Org‑wide Pipelines” dashboard 

Whether you pull via OData or REST, the winning pattern we see with customers: 

Inventory layer (REST) 

All pipelines across projects, with ownership tags, criticality, deployment targets, agent type (hosted/self‑hosted). 

Runs/health layer (OData or REST) 

Success rate, duration (P50/P90), queue time, run frequency, last successful run. 

Governance layer (REST + Graph) 

Service connections used, protected resources/checks, approvals, who can run what, PAT vs. OIDC usage, self‑hosted agent pools. 

Visualization 

Power BI: roll‑up by Project → Repository → Pipeline, slice by YAML/Classic, environment, team, and last activity.
